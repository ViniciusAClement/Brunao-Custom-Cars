0. CONTEXTO (ESTADO ATUAL)
--------------------------------------------------------------------------------
Frontend (Vite + React 19, pasta frontend/ na raiz do repositório):
  - Navegação por estado local; login/logout via API (JWT em sessionStorage); módulo src/api.
  - Peças/funcionários/clientes no painel ainda em memória local (integração C/D pendente).
  - CLIENTE e FUNCIONARIO: telas placeholder; GERENTE: painel admin.
  - Demo API: gerente@empresa.com / senha123.

Backend (Spring Boot 3.2, Java 17):
  - API REST com JWT; H2 em memória; usuário seed: gerente@empresa.com / senha123.
  - Registro de usuários: POST /auth/register exige GERENTE autenticado (não é cadastro público).
  - Modelo de produto: Product (name, description, price, categories, cars) — não bate com
    "peça" do front (nome, marca, modelo, ano, preco) sem mapeamento ou mudança de contrato.
  - jwt.secret está em Base64 válido (32 bytes) compatível com JwtUtil.getSignInKey().
  - PUT/DELETE em /products, /clients, /funcionarios: apenas GERENTE (GET/listagem ainda para
    qualquer usuário autenticado).

1. REQUISITOS FUNCIONAIS (INTEGRAÇÃO)
--------------------------------------------------------------------------------
RF1  O usuário deve autenticar-se contra POST /auth/login com email e senha reais do backend.
RF2  O token JWT deve ser armazenado de forma segura no cliente (preferir memória + refresh
     ou httpOnly cookie em evolução; evitar localStorage se política de segurança exigir).
RF3  Todas as requisições autenticadas devem enviar: Authorization: Bearer <token>.
RF4  Logout deve chamar POST /auth/logout com o mesmo Bearer token (blacklist no servidor).
RF5  Listagem de produtos na home deve vir de GET /products (ou endpoint acordado), não de
     estado vazio/mock.
RF6  CRUD de "peças" no painel gerente deve usar a API de Product (ou novos endpoints/DTOs)
     com mapeamento explícito entre modelo UI e modelo API.
RF7  CRUD de funcionários: POST/PUT/DELETE /funcionarios exigem GERENTE; GET ainda para
     qualquer autenticado (ajustar se quiser restringir listagem).
RF8  CRUD de clientes: POST/PUT/DELETE /clients exigem GERENTE; GET ainda para qualquer
     autenticado (ajustar se necessário).
RF9  Cadastro "Criar conta" na loja: ou implementar fluxo público no backend (novo endpoint)
     ou redirecionar para política real (apenas gerente cria usuários) e ajustar copy da UI.
RF10 Área do cliente logado: definir telas e endpoints (ex.: só leitura de produtos, pedidos
     futuros); corrigir roteamento após login para não exibir painel de gerente.

================================================================================
2. REQUISITOS NÃO FUNCIONAIS
--------------------------------------------------------------------------------
RNF1 CORS: definir origin exata do Vite (ex.: http://localhost:5173) em vez de "*" com
     credentials, quando possível.
RNF2 Variável de ambiente no front: VITE_API_BASE_URL apontando para http://localhost:8080
     (ou porta real do Spring).
RNF3 Tratamento de 401: limpar sessão/token e redirecionar para login; opcional refresh token
     (não existe hoje no backend).
RNF4 Mensagens de erro: mapear respostas do GlobalExceptionHandler (strings) ou evoluir API
     para JSON de erro consistente.
RNF5 Alinhar credenciais de demonstração: ou mudar DataInitializer para gerente@brunao.com ou
     atualizar textos do front para gerente@empresa.com.
RNF6 Documentar contratos (campos obrigatórios) de LoginRequest, RegisterRequest, Product*,
     Client*, Funcionario* para o time de front.

6. RISCOS / DÍVIDAS TÉCNICAS
--------------------------------------------------------------------------------
- Blacklist de tokens em memória: reinício do servidor invalida a lista; aceitável só em dev.
- H2 create-drop: dados somem ao reiniciar; integração local previsível mas não persistente.
- Dois arquivos App (App.jsx ativo, App.tsx morto): evitar confusão removendo ou unificando.
- Smoke test sem token: GET / (texto) ou GET /health (JSON {"status":"UP"}).
